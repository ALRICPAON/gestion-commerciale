## 13. Refonte statuts achats — ETAT RETENU

### Statuts métier utiles

- `ordered` = Commande / BL à préparer
- `received` = Réception validée, stock créé
- `closed` = Clôture future après lettrage / facture
- `cancelled` = Achat abandonné

### Statuts à ne plus utiliser manuellement

- `draft` = à supprimer progressivement du flux standard achat manuel
- `receiving` = inutile dans le flux actuel

### Règles importantes

- `received` ne doit jamais être choisi à la main
- `closed` ne doit jamais être choisi à la main
- `closed` sera déclenché plus tard par module facture / lettrage
- `cancelled` reste manuel avant verrouillages

### Mise à jour importante

La création achat part désormais en `ordered` pour rester cohérente avec le front achat simplifié.

## 15. Réception métier — VALIDE MAIS AVEC POINT TECHNIQUE RESTANT

### Bug initial confirmé

Lors de la validation réception :

- `received_quantity = 0`
- `received_colis = 0`
- `received_pieces = 0`

Donc :

- aucun lot créé
- aucun mouvement stock créé
- lignes non réceptionnées
- mais achat passait quand même en `received`

### Cause exacte

Le backend utilisait uniquement :

- `received_quantity`
- `received_colis`
- `received_pieces`

Alors que dans le flux réel :

- l’utilisateur saisissait surtout `ordered_*`
- `received_*` restait vide

### Décision V2 validée

Pour chaque ligne :

- si `received_* > 0` → utiliser `received_*`
- sinon → utiliser `ordered_*` en fallback

Puis :

- remplir `received_*`
- créer lot
- créer mouvement stock
- mettre à jour la ligne

### Règle critique

Si aucune ligne réceptionnable :

- `createdLots = 0`

Alors :

- refus validation
- ne pas passer achat en `received`
- message : `Aucune quantite receptionnee`

### Résultat test réel

Validé :

- fallback `ordered -> received` OK
- création lot OK
- création mouvement stock OK
- mise à jour ligne OK
- achat passe bien en `received`

### Nouveau bug identifié ensuite

Erreur réelle rencontrée :

- `duplicate key value violates unique constraint "lots_store_lot_code_unique"`

Cause retenue :

- génération de `lot_code` pas assez unique

Décision technique retenue :

- garder la contrainte SQL unique
- corriger le code applicatif
- ajouter un suffixe dérivé de `line.id`

Exemple retenu :

```js
const shortLineId = String(line.id).replace(/-/g, "").slice(0, 6).toUpperCase();
const lotCode = `${pluPart}-${yy}${ddd}-${supplierPart}-${shortLineId}`;
```

## 24. Nouveau bloc stratégique V2 — MODULE VENTES / SORTIES

### 24.1 Décision produit validée

La V2 doit créer un vrai **module Ventes / Sorties**.

Ce module sera l’équivalent métier du module Achats, mais côté sortie de stock.

Il servira à gérer :

- ventes issues d’inventaire
- ventes manuelles
- rétrocessions inter-rayons
- pertes / casse
- ajustements sortants

### 24.2 Pourquoi ce module est nécessaire

L’inventaire V1 appliquait directement des corrections de stock.

En V2, ce n’est plus la bonne logique.

Décision retenue :

- ne jamais corriger le stock directement sans document métier
- toute sortie doit passer par un document traçable
- la modification du stock doit se faire seulement à la validation du document

### 24.3 Règle fondatrice V2

Le stock change uniquement via :

- `stock_movements`
- décrément FIFO des `lots`
- recalcul `stock_summary`

Donc le module ventes / sorties devra appliquer exactement cette règle à la validation.

### 24.4 Types de documents de sortie prévus

Valeurs métier retenues :

- `inventory_sale` = vente déduite d’un inventaire
- `manual_sale` = vente manuelle
- `transfer_out` = rétrocession inter-rayon sortante
- `waste` = casse / perte
- `adjustment_out` = correction sortante exceptionnelle

### 24.5 Statuts documents ventes

Statuts retenus :

- `draft`
- `validated`
- `cancelled`

Règle :

- tant que `draft` → modifiable / supprimable
- une fois `validated` → verrouillé, correction via nouveau document et non par écrasement

## 26. Import CA / Inventaire — règle TRAD / LS retenue pour la V2

### 26.1 Base V1 à reprendre

La V1 import CA sait déjà :

- lire le fichier caisse
- extraire l’EAN
- agréger par EAN
- stocker `{ ca, qty }`

Cette logique est une bonne base fonctionnelle à reprendre côté backend V2.

### 26.2 Règle TRAD retenue

Pour les articles TRAD :

- on part du `CA TTC`
- on utilise le `PV TTC réel` si disponible
- sinon un prix de vente conseillé / fallback défini plus tard
- quantité vendue = `CA TTC / PV TTC`

### 26.3 Règle LS retenue — TRÈS IMPORTANTE

Pour les articles LS :

- on ne calcule **pas** la quantité vendue avec `CA / PV`
- on utilise la **quantité UVC** du fichier caisse
- on reconstruit le prix de vente unitaire pour la ligne

Formule retenue :

- `sold_quantity = qty`
- `unit_sale_price_ttc = ca / qty` si `qty > 0`
- `line_total_ttc = ca`

Raison métier :

- la majorité des articles LS n’ont pas toujours un PV renseigné dans le stock
- le fichier caisse contient déjà la quantité UVC réellement vendue
- il faut donc s’appuyer sur la quantité caisse et non sur un calcul approximatif via le stock

### 26.4 Conséquence V2

L’import CA devra générer des lignes de vente correctes même si `pv_ttc_real` n’est pas renseigné pour une grande partie du LS.

## 53. Annulation validee avec rollback stock - TRANSFORMATION + FABRICATION VALIDEES (2026-05-10)

### 53.1 Decision metier majeure

Decision validee :

Une operation validee ne doit pas etre supprimee brutalement.

Pour corriger une erreur apres validation, il faut une **annulation metier** qui restaure proprement :

- les lots
- les mouvements de stock
- `stock_summary`
- la tracabilite

### 53.2 Transformation validee - Annulation validee

Route ajoutee :

- `POST /api/transformations/:id/cancel-validated`

Fonctionnel valide en reel :

- bouton visible sur une transformation validee :
  - `Annuler transformation validee`
- confirmation utilisateur
- restauration des lots sources
- suppression du lot cible si non consomme
- suppression des mouvements lies
- recalcul du stock
- passage de la transformation en `cancelled`

Protection validee :

- si le lot cible a deja ete consomme / vendu / transforme, l'annulation est refusee

Resultat metier :

- rollback transformation propre
- stock restaure correctement
- mouvements coherents
- pas de suppression brute

### 53.3 Fabrication validee - Annulation validee

Route ajoutee :

- `POST /api/fabrications/:id/cancel-validated`

Fonctionnel valide :

- bouton visible sur une fabrication validee :
  - `Annuler fabrication validee`
- confirmation utilisateur
- restauration des lots ingredients consommes
- suppression du lot produit fini si non consomme
- suppression des mouvements lies
- remise des lignes fabrication en `pending`
- conservation de `used_quantity` pour garder la trace de ce qui avait ete tente
- recalcul du stock des ingredients et du produit fini
- passage de la fabrication en `cancelled`

Protection validee :

- si le lot fabrique a deja ete consomme / vendu / transforme, l'annulation est refusee

### 53.4 Regle generale retenue pour tout le projet

Pour tous les modules stock futurs :

- `draft` / `in_progress` -> suppression directe possible selon regles metier
- `validated` -> annulation metier obligatoire
- `cancelled` -> document conserve comme trace
- jamais de suppression brutale d'un document valide ayant impacte le stock

### 53.5 Importance metier

Cette regle rend le systeme beaucoup plus fiable :

- correction possible en cas d'erreur terrain
- coherence stock preservee
- tracabilite conservee
- comportement proche d'un vrai ERP metier
